## 전제

- **청중**: 절대 다수가 개발자. 1일차(비개발자 포함 입문)와 달리 용어 풀어쓰기를 줄이고 원리·실무 밀도를 올린다. 단 1일차를 수료한 비개발자도 따라올 수 있게 1일차 비유(나무 가지, 단체사진, 세이브 포인트)는 회수해서 쓴다.
- **선행 사실**: 수강생은 1일차에 **first-contributions 저장소에 기여하며 이미 첫 PR을 보냈다**. 즉 fork → clone → branch → commit → push → PR 전 과정을 "따라 하며" 한 번 경험한 상태다.
- **위치**: 입문 트랙의 완결편. 2일차의 관통 서사는 **"어제 따라 한 모든 동작의 실체를 공개하고, 오늘 배운 워크플로우로 앞으로 무엇을 만들지 스스로 정한다"**. (실제 PR 실행은 4일차)
- **경계** (중복 방지 원칙):
  - 코드 리뷰 왕복·리뷰 문화 → 3일차 몫. s2는 "PR을 만들어 보내는 것"까지.
  - 포트폴리오 완성(README 다듬기 심화, 프로필, CI 배지) → 4일차 몫. s2는 "기록이 자산"이라는 동기부여와 README 기본기까지.

### 2막 · Git 원리 심화

키워드: Git 원리 심화, 버전 관리, reset·revert·stash

| # | 슬라이드 | 내용 |

| 5 | 커밋의 정체 | 스냅샷 + SHA 해시. 해시는 **내용 + 부모 커밋 + 작성자 + 시간 + 메시지**를 반영한 지문(fingerprint) — 1일차 git log의 `a1b2c3d`가 무엇이었는지 공개 |
| 6 | 브랜치의 정체 | **브랜치는 커밋을 가리키는 이름표.** 결론: "브랜치는 공짜다, 마음껏 만들어라" (1일차 '가지' 비유의 실체. 내부 구현 디테일은 다루지 않음) |
| 7 | HEAD의 이동 | checkout이 실제로 하는 일 = HEAD 포인터 옮기기. 다음 장(reset)의 기반 |

| 9 | 되돌리기 2: stash | 커밋하기엔 애매한 작업 잠깐 치워두기. 시나리오: 작업 중 급한 브랜치 전환 |

### 4막 · 협업과 프로세스

키워드: 협업 방법, 프로젝트 관리 프로세스, GitHub 권한 관리, origin·upstream

| # | 슬라이드 | 내용 |
|---|---------|------|
| 13 | 권한이 협업 방식을 가른다 | 저장소 권한(read/write/admin). write 있으면 collaborator(팀, 브랜치 직접 push), 없으면 **fork**(내 사본에서 작업 후 PR). "fork를 왜 하는가"의 답 = 권한. **어제 first-contributions에서 fork 버튼을 눌렀던 이유가 바로 이것** (write 권한이 없었으니까) |
| 14 | origin과 upstream | fork 실습 최대 혼동 지점. **내 fork = origin, 원본 저장소 = upstream, PR = origin의 브랜치를 upstream에 제안하는 것.** 어제 clone한 주소가 origin(내 fork)이었다는 사실 공개. `git remote add upstream`과 upstream에서 최신 받아오기(fetch/pull)까지. 다이어그램 필수 (로컬 ↔ origin ↔ upstream 3자 구도) |
| 15 | GitHub 프로젝트 관리 루프 | Issue → branch → commit → PR → merge. 이슈 번호(`#12`)로 연결되는 추적성. 실무의 일감 관리가 이 루프 위에서 돈다 |
| 16 | PR이라는 문서 | 제목·설명·연결된 이슈. 리뷰어가 읽는 순서로 쓰기. **1막의 훅 질문("어제 만든 PR은 무슨 일이었나")이 여기서 완전히 풀린다.** (리뷰 왕복 자체는 3일차 예고) |
