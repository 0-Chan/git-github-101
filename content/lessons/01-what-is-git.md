## 이런 경험, 있으시죠?

중요한 문서를 작성하다가 이런 파일 목록을 만들어 본 적 있나요?

```
보고서_최종.docx
보고서_최종_수정.docx
보고서_최종_진짜최종.docx
보고서_최종_이걸로.docx
```

이 방식에는 세 가지 문제가 있습니다.

- **어느 게 진짜 최신인지 알 수 없습니다.** 파일 이름은 거짓말을 합니다.
- **"수정 전으로 돌아가고 싶다"가 안 됩니다.** `_진짜최종`에서 무엇이 바뀌었는지 아무도 기억하지 못합니다.
- **여럿이 함께 작업하면 재앙이 됩니다.** 같은 파일을 두 사람이 고치면 한 사람의 작업이 조용히 사라집니다.

## 버전 관리: 기록하고, 되돌아가는 기술

**버전 관리 시스템**(VCS, Version Control System)은 이 문제를 해결하는 도구입니다. 파일을 복사해서 쌓아두는 대신, **변경의 역사 자체를 기록**합니다.

- 언제, 누가, 무엇을, **왜** 바꿨는지가 자동으로 남습니다.
- 어느 시점으로든 안전하게 되돌아갈 수 있습니다 — 파일 이름을 바꿀 필요 없이요.
- 여러 사람이 같은 프로젝트를 동시에 작업해도 서로의 변경이 사라지지 않습니다.

말하자면 프로젝트 전용 **타임머신 + 항해 일지**입니다.

## Git이란?

**Git**은 세계에서 가장 널리 쓰이는 버전 관리 도구입니다. 2005년 리누스 토르발스(리눅스를 만든 사람)가 리눅스 개발을 위해 만들었고, 지금은 사실상 모든 소프트웨어 팀의 표준이 되었습니다.

이름 때문에 어렵게 느껴지지만 본질은 단순합니다 — **"지금 이 상태를 기억해 둬"라고 말할 수 있게 해주는 프로그램**입니다. 그리고 코드만이 아니라 텍스트로 된 것이라면 무엇이든(문서, 설정, 원고…) 관리할 수 있습니다.

## 세 가지 핵심 개념

이 튜토리얼 전체에서 계속 만나게 될 세 단어입니다. 지금은 감만 잡으면 충분합니다.

### 저장소 (Repository)

프로젝트의 모든 파일과 **변경 이력 전체**가 담기는 공간입니다. 내 컴퓨터에 있으면 **로컬 저장소**, GitHub 같은 서버에 있으면 **원격 저장소**라고 부릅니다. 프로젝트 전용 타임머신 보관함이라고 생각하세요.

### 커밋 (Commit)

특정 시점의 상태를 저장하는 단위 — **게임의 세이브 포인트**입니다. 커밋을 만들어 두면 그 시점으로 언제든 돌아올 수 있고, 각 커밋에는 "무엇을 왜 바꿨는지" 메모(커밋 메시지)가 붙습니다.

### 브랜치 (Branch)

원본에 영향을 주지 않는 **독립적인 작업 공간**입니다. 본편은 그대로 두고 초안을 따로 써보는 것과 같습니다. 실험이 성공하면 본편에 합치고(머지), 실패하면 버리면 됩니다.

## Git과 GitHub는 다릅니다

처음 배울 때 가장 많이 혼동하는 부분입니다. 이름은 비슷하지만 둘은 서로 다른 것입니다.

|  | Git | GitHub |
| --- | --- | --- |
| 정체 | 내 컴퓨터에서 실행되는 **프로그램(도구)** | Git 저장소를 올려두고 공유하는 **웹 서비스** |
| 비유 | 한글/워드 같은 프로그램 | 구글 드라이브 같은 클라우드 |
| 인터넷 필요? | 필요 없음 (전부 내 컴퓨터에서 동작) | 필요함 |
| 없어도 되나? | GitHub 없이 Git만 써도 됨 | GitHub은 Git 없이는 의미가 없음 |

GitHub 말고도 GitLab, Bitbucket 같은 비슷한 서비스가 있습니다. 이 강의에서는 레슨 02~08에서 **Git**을, 레슨 09부터 **GitHub**을 다룹니다.
