## Pull Request란?

**Pull Request(PR)**는 내가 작업한 브랜치의 변경사항을 다른 브랜치(보통 `main`)에 반영해달라고 요청하는 것입니다.

단순히 코드를 합치는 것이 아니라, **팀원들이 코드를 검토하고 피드백을 주고받는** 협업의 핵심 과정입니다.

## 코드 리뷰 워크플로우

실제 팀에서는 보통 다음과 같은 흐름으로 작업합니다:

### 1단계: 브랜치 생성
```bash
git checkout -b feature/new-login
```

### 2단계: 작업 후 커밋
```bash
git add .
git commit -m "feat: 로그인 페이지 UI 구현"
```

### 3단계: 원격 저장소에 푸시
```bash
git push origin feature/new-login
```

### 4단계: Pull Request 생성
GitHub에서 **Compare & pull request** 버튼을 클릭해 PR을 만듭니다.

### 5단계: 코드 리뷰
팀원들이 코드를 검토하고 댓글로 피드백을 남깁니다.

### 6단계: 수정 및 재푸시
피드백을 반영해 코드를 수정하고 같은 브랜치에 다시 푸시하면 PR에 자동으로 반영됩니다.

### 7단계: 머지
승인을 받으면 `main` 브랜치에 머지합니다.

## PR 작성 모범 사례

### 제목을 명확하게 작성하세요
- 나쁜 예: `수정`
- 좋은 예: `feat: 로그인 페이지 유효성 검사 추가`

### 변경 내용을 자세히 설명하세요
PR 본문에는 다음 내용을 포함하는 것이 좋습니다:
- **무엇을 변경했나요?**: 변경사항 요약
- **왜 변경했나요?**: 변경의 이유 또는 배경
- **어떻게 테스트했나요?**: 동작을 확인한 방법
- **스크린샷**: UI 변경이라면 이미지 첨부

### 작은 단위로 PR을 만드세요
한 PR에 너무 많은 변경사항이 담기면 리뷰하기 어렵습니다. 하나의 기능 또는 버그 수정 단위로 PR을 만드는 것이 좋습니다.

### 리뷰어를 배려하세요
코드에 설명이 필요한 부분이 있다면 PR 댓글로 먼저 설명을 달아주면 리뷰어가 맥락을 이해하는 데 도움이 됩니다.
